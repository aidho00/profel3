Fundamentals of Database Systems 01 Overview

The Database Environment

A database doesn't exist in isolation. It lives within an environment of hardware, software, data, procedures, and people all working together.

Think of it like an ecosystem: every component depends on the others.

Fundamentals of Database Systems 01 Environment Layers

Layers of the DB Environment

🖥️ Hardware Layer

Servers, storage (HDD/SSD), RAM, network infrastructure, backup devices

⚙️ Software Layer

OS, DBMS (MySQL), utilities, application programs, middleware

🌐 Network Layer

LAN, internet, protocols (TCP/IP), firewalls that connect users to the DB

👥 User Layer

End users, developers, DBAs accessing through various interfaces

Fundamentals of Database Systems 02 Architecture

What is Client-Server?

A model where one machine (server) provides services, and other machines (clients) request those services over a network.

Client

Sends requests. Displays results. Your browser, app, or phpMyAdmin.

Server

Processes requests. Manages data. MySQL running in the background.

Fundamentals of Database Systems 02 1-Tier

1-Tier (Single Machine)

Everything on one computer:

  • User interface + business logic + database all on same machine
  • Simple but not scalable
  • Good for: personal projects, testing, learning
  • Bad for: multiple users, production systems

📌 This is what we do in class with WAMP running locally.

Fundamentals of Database Systems 02 2-Tier

2-Tier (Client-Server)

Client talks directly to database server:

  • Client: handles UI and some logic
  • Server: handles data storage and queries
  • Connected via network (LAN or internet)
  • Good for: small-medium businesses, office apps

📌 Example: A desktop inventory app connecting to a shared MySQL server in the office.

Fundamentals of Database Systems 02 3-Tier

3-Tier (Web Architecture)

Client → Application Server → Database:

  • Client (browser): only displays UI
  • App server (PHP/Node/Java): handles business logic
  • Database server (MySQL): stores and retrieves data
  • Most scalable and secure architecture

📌 This is how Facebook, Shopee, and your school's enrollment system work.

Fundamentals of Database Systems 03 Query Lifecycle

What Happens When You Query?

When you type a SQL command in phpMyAdmin and hit "Go", here's what MySQL does behind the scenes:

  • Step 1: Connection — Client connects to MySQL server
  • Step 2: Parsing — MySQL reads and validates your SQL syntax
  • Step 3: Optimization — MySQL finds the fastest way to execute
  • Step 4: Execution — The query runs against the data
  • Step 5: Result — Data is returned to the client
Fundamentals of Database Systems 03 Step 1

Connection Manager

MySQL listens on port 3306 by default. When a client connects, MySQL:

  • Authenticates (username + password)
  • Checks permissions (what can this user do?)
  • Assigns a thread for this connection
  • Allocates memory for the session

Default login: username "root", password "" (empty) in WAMP.

Fundamentals of Database Systems 03 Step 2

SQL Parser

The parser reads your SQL and checks:

  • Syntax check: Is the SQL grammatically correct?
  • Semantic check: Do the tables/columns exist?
  • Permission check: Does this user have access?
-- This is valid SQL:
SELECT name FROM students;

-- This will fail parsing:
SELCET name FORM students; -- typos!
Fundamentals of Database Systems 03 Step 3

Query Optimizer

MySQL's optimizer finds the FASTEST way to get your results. It considers:

  • Which indexes to use (shortcuts to find data faster)
  • The order to join tables
  • Whether to scan the full table or use an index

The optimizer is why databases are FAST. It's like GPS finding the quickest route.

Fundamentals of Database Systems 03 Steps 4 & 5

Execution & Results

⚡ Execution Engine

Reads data from disk/memory. Applies filters. Performs JOINs. Computes aggregates. Sorts results.

📤 Result Set

Formatted results sent back to the client. phpMyAdmin displays them as a nice HTML table.

All 5 steps happen in milliseconds for simple queries. Complex queries on large data may take seconds.

Fundamentals of Database Systems Part 04

Database Languages

DDL, DML, DCL, and TCL

Fundamentals of Database Systems 04 SQL Categories

SQL is Not Just One Language

SQL is divided into sub-languages, each with a different purpose:

DDL

Data Definition Language — CREATE, ALTER, DROP (structure)

DML

Data Manipulation Language — SELECT, INSERT, UPDATE, DELETE (content)

DCL

Data Control Language — GRANT, REVOKE (permissions)

TCL

Transaction Control Language — COMMIT, ROLLBACK (transactions)

Fundamentals of Database Systems 04 DDL

DDL: Data Definition Language

Defines the STRUCTURE of the database. Creates and modifies tables.

-- Create a table
CREATE TABLE students (
  id INT PRIMARY KEY,
  name VARCHAR(100),
  course VARCHAR(50)
);

-- Modify a table
ALTER TABLE students ADD gpa DECIMAL(3,2);

-- Delete a table
DROP TABLE students;
Fundamentals of Database Systems 04 DML

DML: Data Manipulation Language

Works with the CONTENT (data) inside tables. The most-used category.

-- Read data
SELECT * FROM students;

-- Add data
INSERT INTO students VALUES (1, 'Maria', 'BSIS');

-- Change data
UPDATE students SET course = 'BSIT' WHERE id = 1;

-- Remove data
DELETE FROM students WHERE id = 1;
Fundamentals of Database Systems 04 DCL & TCL

DCL & TCL

DCL: Control

-- Give access
GRANT SELECT ON students
TO 'teacher';

-- Remove access
REVOKE DELETE ON students
FROM 'teacher';

TCL: Transactions

-- Start transaction
START TRANSACTION;
UPDATE balance...

-- Save or undo
COMMIT;
ROLLBACK;

Fundamentals of Database Systems 05 ODBC / JDBC

Standard Interfaces: ODBC & JDBC

ODBC

Open Database Connectivity. A universal API that lets any application connect to any database regardless of the DBMS. Works across languages.

JDBC

Java Database Connectivity. Java's standard API for connecting to databases. Every Java app uses JDBC to talk to MySQL, Oracle, etc.

These standards mean you learn SQL once and it works everywhere. The API handles the translation.

Fundamentals of Database Systems 05 MySQL Internals

Inside MySQL: Storage Engines

MySQL has "engines" that handle HOW data is physically stored. The most important:

EngineFeaturesUse Case
InnoDB ⭐Transactions, foreign keys, crash recoveryDefault. Most applications.
MyISAMFast reads, no transactionsRead-heavy, legacy systems
MEMORYStored in RAM, extremely fastTemporary data, caching

We'll always use InnoDB. It's the default and most reliable.
