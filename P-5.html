Fundamentals of Database Systems 01 Domain

What is a Domain?

A domain is the set of all possible valid values for an attribute. Think of it as a "value pool."

AttributeDomainExample Values
StudentIDFormat: YYYY-NNN2024-001, 2024-002
NameStrings up to 100 chars"Maria Santos"
Course{"BSIS", "BSIT", "BSCS"}"BSIS"
GPADecimal 1.00 to 5.001.50, 2.00
Fundamentals of Database Systems 01 Domain Rules

Domain Constraints

  • Atomic: Each domain value must be indivisible (no lists, no sets)
  • Typed: All values in a domain share the same data type
  • Named: Domains should be meaningful (not just "VARCHAR")
  • Comparable: Only attributes from the same domain can be meaningfully compared

Example: Comparing StudentID with GPA makes no sense — different domains!

Fundamentals of Database Systems 01 Relation Schema

Relation Schema (Blueprint)

A relation schema defines the NAME and ATTRIBUTES of a relation. Written as:

STUDENTS(StudentID, Name, Course, GPA)
  • Relation name: STUDENTS
  • Attributes: StudentID, Name, Course, GPA
  • Primary key: underlined (StudentID)
  • Degree: 4 (four attributes)
Fundamentals of Database Systems 01 Multiple Schemas

Database Schema Example

-- Database Schema for a School System

STUDENTS(StudentID, Name, Course, GPA)
COURSES(CourseCode, CourseName, Units, Department)
ENROLLMENTS(EnrollID, StudentID, CourseCode, Semester, Grade)
FACULTY(FacultyID, Name, Department, Rank)

The underlined attribute is the primary key. Multiple schemas together form a database schema.

Rules that every valid relation must follow

Fundamentals of Database Systems 02 Property 1

Each Tuple is Unique

No two rows can be identical. Every row must be distinguishable through its primary key.

✅ Valid

IDName
001Maria
002Maria

Same name but different ID = OK

❌ Invalid

IDName
001Maria
001Maria

Duplicate ID = VIOLATION

Fundamentals of Database Systems 02 Property 2

Attribute Values are Atomic

Each cell contains ONE and ONLY ONE value. No lists, no arrays, no nested structures.

✅ Atomic

IDPhone
00109171234567
00109281234567

Separate rows for each phone

❌ Not Atomic

IDPhone
0010917..., 0928...

Multiple values in one cell

Fundamentals of Database Systems 02 Property 3

Tuple Order is Irrelevant

Rows have no inherent order. The same data in any row arrangement is the same relation.

These two are THE SAME relation:

{Maria, Juan, Ana} = {Ana, Maria, Juan} = {Juan, Ana, Maria}

If you need a specific order, use ORDER BY in SQL. The table itself has no order.

Fundamentals of Database Systems 02 Property 4

Attribute Order is Irrelevant

Columns have no inherent order. You reference them by NAME, not position.

Same Relation

IDNameGPA
001Maria1.5

Also Same Relation

NameGPAID
Maria1.5001

In SQL: SELECT Name, GPA, ID produces the same data as SELECT ID, Name, GPA.

Fundamentals of Database Systems 02 Property 5

Attribute Names are Unique

Within a relation, no two attributes can have the same name. Each column must be distinguishable.

✅ Valid

StudentName, FatherName, MotherName

❌ Invalid

Name, Name, Name (ambiguous!)

Fundamentals of Database Systems 02 Summary

5 Properties of Relations

  • 1. Each tuple is unique (no duplicate rows)
  • 2. Attribute values are atomic (one value per cell)
  • 3. Tuple order is irrelevant (rows have no sequence)
  • 4. Attribute order is irrelevant (columns by name, not position)
  • 5. Attribute names are unique (no duplicate column names)
Fundamentals of Database Systems Part 03

NULL Values

The meaning of "unknown" in databases

Fundamentals of Database Systems 03 NULL

What is NULL?

NULL represents a missing, unknown, or not applicable value. It is NOT zero. It is NOT an empty string.

ValueMeaningExample
NULLUnknown / Not applicableStudent hasn't taken the exam yet
0A real numeric valueStudent scored zero on the exam
""An empty string valueStudent left the field blank

NULL ≠ 0 ≠ "" — these are THREE different things!

Fundamentals of Database Systems 03 Types of NULL

Three Types of NULL

Unknown

Value exists but we don't know it yet. (Student's final grade before exam)

Not Applicable

The attribute doesn't apply. (Maiden name for a male student)

Withheld

Value exists but user chose not to provide. (Phone number declined)

Fundamentals of Database Systems 03 NULL Problems

Why NULL is Tricky

  • NULL = NULL is NOT true (even two unknowns can't be compared)
  • NULL + 5 = NULL (any math with NULL = NULL)
  • WHERE column = NULL doesn't work (use IS NULL instead)
  • COUNT(*) counts NULLs, COUNT(column) skips NULLs

We'll deal with NULLs a LOT when writing SQL. For now, just understand the concept.

Fundamentals of Database Systems 04 What Is It?

Relational Algebra

A set of mathematical operations on relations (tables) that produce new relations. It's the THEORY behind SQL.

Think of it this way:

Relational Algebra = the math theory
SQL = the practical language based on that theory

Fundamentals of Database Systems 04 Selection σ

Selection (σ): Filter Rows

Picks rows that satisfy a condition. Like WHERE in SQL.

-- Relational Algebra:
σCourse='BSIS'(STUDENTS)

-- SQL equivalent:
SELECT * FROM STUDENTS WHERE Course = 'BSIS';

Result: Only rows where Course = 'BSIS'

Fundamentals of Database Systems 04 Projection π

Projection (π): Pick Columns

Selects specific columns from a relation. Like listing columns in SELECT.

-- Relational Algebra:
πName, GPA(STUDENTS)

-- SQL equivalent:
SELECT Name, GPA FROM STUDENTS;

Result: Only the Name and GPA columns (removes duplicates in formal algebra)

Fundamentals of Database Systems 04 Combined

Combining Selection + Projection

Get names of BSIS students with GPA better than 1.50:

-- Relational Algebra:
πName(σCourse='BSIS' AND GPA<1.50(STUDENTS))

-- SQL equivalent:
SELECT Name
FROM STUDENTS
WHERE Course = 'BSIS' AND GPA < 1.50;
Fundamentals of Database Systems 04 Join ⋈

Join (⋈): Combine Tables

Combines rows from two tables based on a matching condition (shared key).

-- Relational Algebra:
STUDENTS ⋈STUDENTS.CourseCode=COURSES.CourseCode COURSES

-- SQL equivalent:
SELECT *
FROM STUDENTS
JOIN COURSES ON STUDENTS.CourseCode = COURSES.CourseCode;

We'll master JOINs in Week 7. For now, understand the concept.

How we document and describe relations

Fundamentals of Database Systems 05 Schema vs State

Schema vs. State (Instance)

Schema (Structure)

The DEFINITION. Doesn't change often. Like a form template.

STUDENTS(ID, Name, GPA)

State (Instance)

The CURRENT DATA. Changes with every insert/update/delete.

{(001,"Maria",1.5), (002,"Juan",1.75)}

Schema = blueprint (rarely changes). State = current contents (always changing).

Fundamentals of Database Systems 05 Notation

How to Write Relation Schemas

-- Standard notation (primary key underlined):

STUDENTS(StudentID, Name, Course, GPA)

COURSES(CourseCode, CourseName, Units)

ENROLLMENTS(EnrollID, StudentID, CourseCode, Grade)

-- Underlined = Primary Key
-- Italic/colored = Foreign Key (references another table)
Fundamentals of Database Systems 05 Formal Notation

Formal Mathematical Notation

  • R(A₁, A₂, ..., Aₙ) — Relation R with n attributes
  • r(R) — Current state (instance) of relation R
  • t ∈ r(R) — Tuple t is a member of relation state r
  • t[Aᵢ] — Value of attribute Aᵢ in tuple t
  • dom(Aᵢ) — Domain of attribute Aᵢ

You won't use this daily, but textbooks and exams use it. Know what it means.

Fundamentals of Database Systems 05 Example

Reading Formal Notation

Given: STUDENTS(StudentID, Name, Course, GPA)

t = (2024-001, "Maria Santos", "BSIS", 1.50)

  • t[Name] = "Maria Santos"
  • t[GPA] = 1.50
  • dom(GPA) = {x | 1.00 ≤ x ≤ 5.00}
  • Degree of STUDENTS = 4
