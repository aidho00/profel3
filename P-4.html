Fundamentals of Database Systems 01 Today

What is a Data Model?

A data model is a conceptual framework that defines how data is organized, stored, and manipulated in a database system.

Think of it as the "blueprint style" — like how buildings can be Gothic, Modern, or Traditional, databases can follow different models.

Fundamentals of Database Systems 01 Why It Matters

Why Study Data Models?

  • Helps you choose the RIGHT approach for your data
  • Understanding history helps you appreciate modern solutions
  • Job interviews ask about different models
  • Different projects need different models (not everything is relational)

Fundamentals of Database Systems 01 Characteristics

Hierarchical Model Features

✅ Advantages

Fast for hierarchical queries (organizational charts). Simple parent-child navigation. Good performance for specific use cases.

❌ Disadvantages

Rigid structure: can't handle many-to-many. Data redundancy if item belongs to multiple parents. Complex to reorganize.

Example system: IBM IMS (1966). Still used in some legacy banking systems.

Fundamentals of Database Systems 01 Real Example

Hierarchical Model in Real Life

  • 📁 Your computer's file system (folders inside folders)
  • 🏢 Organizational charts (CEO → VP → Manager → Staff)
  • 🌐 XML/HTML documents (nested tags)
  • 📱 Android's View hierarchy (layouts → buttons)

The folder system on your computer IS a hierarchical model!

Fundamentals of Database Systems Part 02

Network Model

More flexible than hierarchical (1960s-70s)

Fundamentals of Database Systems 02 Network Model

Graph Structure

Like hierarchical, but a child can have MULTIPLE parents. Forms a graph/network instead of a tree.

Key Difference

A student can belong to multiple courses. A course can have multiple students. Many-to-many is allowed.

Structure

Records connected by "sets" (like pointers). Navigate by following links between records.

Fundamentals of Database Systems 03 The Game Changer

The Relational Model

Proposed by E.F. Codd in 1970. Data stored in tables (relations). No pointers, no complex navigation. Just simple rows and columns.

Revolutionary idea: Users don't need to know HOW data is stored. They just ASK for what they want using SQL.

Fundamentals of Database Systems 03 Key Principles

Relational Model Rules

  • Each row is unique (identified by a primary key)
  • Column order doesn't matter
  • Row order doesn't matter
  • Each cell holds ONE value (atomic)
  • Tables relate through shared keys (foreign keys)

Fundamentals of Database Systems 03 Why It Won

Why Relational Dominates

  • Simple: Tables are intuitive (like spreadsheets)
  • Flexible: Easy to add columns, tables, relationships
  • Powerful: SQL can answer complex questions in one query
  • Data independent: Change storage without affecting apps
  • Mathematical: Based on set theory and relational algebra

Fundamentals of Database Systems 04 Object-Oriented

Object-Oriented Model (OODBMS)

Stores data as objects (like in Java/Python). Each object has properties and methods.

✅ Good For

Complex data types (multimedia, CAD, scientific). Matches object-oriented programming naturally.

❌ Limitations

No standard query language like SQL. Smaller community. Not widely adopted in business.

Examples: ObjectDB, db4o. Niche use cases only.

Fundamentals of Database Systems 04 Object-Relational

Object-Relational Model (ORDBMS)

The best of both worlds: relational tables + object features (custom types, inheritance, methods).

PostgreSQL is the best example of an ORDBMS.

  • Supports standard SQL (relational)
  • Also supports custom data types, JSON, arrays
  • Table inheritance (a table can extend another)
Fundamentals of Database Systems Part 05

Modern alternatives for modern problems

Fundamentals of Database Systems 05 Why NoSQL?

When Relational Isn't Enough

Some modern apps need what relational databases struggle with:

  • Massive scale: Billions of records, thousands of servers
  • Flexible schema: Data structure changes constantly
  • High speed: Millions of reads/writes per second
  • Unstructured data: JSON, images, social graphs
Fundamentals of Database Systems 05 NoSQL Types

4 Types of NoSQL

📄 Document

Stores JSON/BSON documents. Each document is self-contained. MongoDB, CouchDB

🔑 Key-Value

Simple: a key maps to a value. Blazing fast for lookups. Redis, DynamoDB

📊 Column-Family

Data stored by columns, not rows. Great for analytics. Cassandra, HBase

🕸️ Graph

Nodes and edges. Perfect for relationships. Neo4j, Amazon Neptune

Fundamentals of Database Systems 05 Document DB

Fundamentals of Database Systems 05 Key-Value

Key-Value Model (Redis)

// Simple key → value pairs

GET "user:001:name"
→ "Maria Santos"

Extremely fast (in-memory). Used for caching, sessions, real-time leaderboards.

Fundamentals of Database Systems 06 Reality Check

Most Apps Use Relational

Despite the NoSQL hype:

  • 80%+ of business applications use relational databases
  • MySQL and PostgreSQL dominate job postings
  • NoSQL is for specific use cases, not a replacement
  • Many companies use BOTH (polyglot persistence)

For this course: We focus on Relational (MySQL) because it's the foundation of everything.

Fundamentals of Database Systems 06 Recap

Today's Key Takeaways

  • ✅ Hierarchical: tree structure, rigid (file systems)
  • ✅ Network: graph structure, flexible but complex
  • ✅ Relational: tables + SQL, dominant model (our focus)
  • ✅ NoSQL: Document, Key-Value, Column, Graph (for scale)
  • ✅ Most apps use relational; big apps use polyglot persistence
