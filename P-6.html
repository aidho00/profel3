Fundamentals of Database Systems Part 01

Understanding Keys

How we uniquely identify records

Fundamentals of Database Systems 01 Why Keys?

Why Do We Need Keys?

Keys solve the fundamental problem: How do we tell two similar records apart?

❌ Without Keys

Two "Juan Cruz" enrolled. Which one got the 1.5? Which owes tuition? We can't tell.

✅ With Keys

Student 2024-001 "Juan Cruz" vs 2024-045 "Juan Cruz". Unique IDs solve everything.

Fundamentals of Database Systems 01 Super Key

Super Key

Any set of attributes that can uniquely identify every tuple. Can include extra (unnecessary) attributes.

STUDENTS(StudentID, Name, Email, Course)

Super keys include:

  • {StudentID} ← minimal
  • {Email} ← minimal
  • {StudentID, Name} ← works but Name is extra
  • {StudentID, Name, Email, Course} ← entire row, always unique
Fundamentals of Database Systems 01 Candidate Key

Candidate Key

A minimal super key: remove any attribute and it stops being unique. These are your "nominees" for primary key.

Attribute(s)Unique?Minimal?Candidate Key?
{StudentID}✅ Yes✅ Yes✅ Yes
{Email}✅ Yes✅ Yes✅ Yes
{StudentID, Name}✅ Yes❌ No (Name unnecessary)❌ No
{Name}❌ No (duplicates possible)—❌ No
Fundamentals of Database Systems 01 Primary Key

Primary Key (PK)

The ONE candidate key you CHOOSE as the official identifier. Rules:

  • Unique: No two rows can have the same PK value
  • NOT NULL: Every row MUST have a PK value
  • Immutable: Should rarely (ideally never) change
  • One per table: Only one primary key allowed

Common choice: auto-incrementing integers (1, 2, 3...) or formatted IDs (2024-001)

Fundamentals of Database Systems 01 PK in SQL

Primary Key in MySQL

CREATE TABLE students (
  student_id INT AUTO_INCREMENT,
  name VARCHAR(100) NOT NULL,
  email VARCHAR(100) UNIQUE,
  course VARCHAR(50),
  PRIMARY KEY (student_id)
);
  • AUTO_INCREMENT: MySQL generates 1, 2, 3... automatically
  • PRIMARY KEY: Enforces uniqueness + NOT NULL
Fundamentals of Database Systems 01 Natural vs Surrogate

Natural vs. Surrogate Keys

Natural Key

A real-world attribute: Email, SSN, ISBN, Student ID

✅ Meaningful
❌ Can change (email changes)
❌ May be long

Surrogate Key

System-generated: auto-increment integer, UUID

✅ Never changes
✅ Short & fast
❌ No real meaning

Best practice: Use surrogate keys (AUTO_INCREMENT) as primary keys. Add UNIQUE on natural keys.

Fundamentals of Database Systems 01 Composite Key

Composite Key

A key made of TWO or more columns combined. Used when no single column is unique.

-- ENROLLMENTS: a student can enroll in many courses
-- but the same student + same course = unique pair

CREATE TABLE enrollments (
  student_id INT,
  course_code VARCHAR(10),
  semester VARCHAR(20),
  grade DECIMAL(3,2),
  PRIMARY KEY (student_id, course_code, semester)
);

Foreign Keys

Connecting tables together

Fundamentals of Database Systems 02 Foreign Key

What is a Foreign Key?

A column (or set of columns) in one table that references the primary key of another table. It creates a LINK between tables.

STUDENTS(student_id, name, course)

ENROLLMENTS(enroll_id, student_id, course_code, grade)

student_id in ENROLLMENTS is a Foreign Key → references student_id in STUDENTS

Fundamentals of Database Systems 02 FK in SQL

Foreign Key in MySQL

CREATE TABLE enrollments (
  enroll_id INT AUTO_INCREMENT PRIMARY KEY,
  student_id INT NOT NULL,
  course_code VARCHAR(10) NOT NULL,
  grade DECIMAL(3,2),

  FOREIGN KEY (student_id)
    REFERENCES students(student_id)
    ON DELETE CASCADE
    ON UPDATE CASCADE
);
Fundamentals of Database Systems 02 FK Rules

Foreign Key Rules

  • Rule 1: FK value MUST exist in the referenced table (or be NULL)
  • Rule 2: Can't insert a child record with a non-existent parent
  • Rule 3: Can't delete a parent if children still reference it (unless CASCADE)
  • Rule 4: FK can be NULL (student not yet assigned to a course)

Fundamentals of Database Systems Part 03

Integrity Constraints

Rules that protect your data's quality

Fundamentals of Database Systems 03 Overview

Types of Integrity Constraints

Entity Integrity

Primary key must be unique and NOT NULL. Every row must be identifiable.

Referential Integrity

Foreign key must reference a valid primary key (or be NULL).

Domain Integrity

Values must be within allowed range/type for that column.

Fundamentals of Database Systems 03 Entity Integrity

Entity Integrity

Rule: No primary key attribute may be NULL. Every row must have a valid, unique identifier.

✅ Valid

❌ Violation

PK cannot be NULL!

Fundamentals of Database Systems 03 Referential Integrity

Referential Integrity

Rule: A foreign key must reference an existing primary key value, or be NULL. No "orphan" records.

✅ Valid FK

Enrollment has student_id = 1. Student 1 EXISTS in Students table.

❌ Orphan Record

Enrollment has student_id = 999. Student 999 DOESN'T EXIST. Rejected!

MySQL enforces this automatically when you define FOREIGN KEY constraints.

Fundamentals of Database Systems 03 Domain Integrity

Domain Integrity

Rule: Values must be valid for their defined domain (data type, range, format).

CREATE TABLE students (
  id INT,
  name VARCHAR(100) NOT NULL, -- can't be empty
  gpa DECIMAL(3,2) CHECK(gpa >= 1.00 AND gpa <= 5.00),
  age INT CHECK(age >= 15 AND age <= 80),
  gender ENUM('Male', 'Female', 'Other')
);

Fundamentals of Database Systems 03 Full Example

Complete Table with All Constraints

CREATE TABLE students (
  student_id INT AUTO_INCREMENT,
  name VARCHAR(100) NOT NULL,
  email VARCHAR(100) NOT NULL UNIQUE,
  course ENUM('BSIS','BSIT','BSCS') NOT NULL,
  gpa DECIMAL(3,2) CHECK(gpa >= 1.00 AND gpa <= 5.00),
  status VARCHAR(20) DEFAULT 'Active',
  enrolled_date DATE NOT NULL,
  PRIMARY KEY (student_id)
);
Fundamentals of Database Systems Part 04

Common Mistakes

Pitfalls to avoid when designing keys

Fundamentals of Database Systems 04 Bad Keys

What NOT to Use as Primary Key

  • Name — Duplicates exist (many "Juan Cruz" in Philippines)
  • Phone Number — People change numbers
  • Email — Can change; use as UNIQUE, not PK
  • Address — Changes frequently, too long
  • Composite of name+birthday — Still not guaranteed unique

Best practice: Always use AUTO_INCREMENT INT or a system-generated ID.
